# nikkikoole.github.io / mipolai.com

The source for [mipolai.com](https://mipolai.com) — a small hand-rolled static site
generator in Lua. Write Markdown in `content/`, run the build, and it renders HTML into
`docs/`, which GitHub Pages serves.

## Layout

```
content/     source Markdown + TOML-ish frontmatter (what you edit)
templates/   liluat HTML templates (page shell, head, header/footer, meta tags)
libs/        the Lua deps (markdown, toml, liluat, inspect, watcher)
docs/        BUILD OUTPUT — the published site (Pages serves this /docs folder).
             Regenerated by the build; also holds assets/ (images, fonts, sounds).
tools/       helper scripts (see below)
main.lua     the build: content/ + templates/ -> docs/*.html + sitemap
```

## Develop locally

```sh
lua run.lua
```

Serves `docs/` at **http://localhost:8080** *and* watches for changes, rebuilding on
every save. Edit under `content/`, refresh the browser. Nothing is published until you
commit + push — this is fully local.

Build once without serving:

```sh
lua main.lua
```

## Publish

Pages deploys from the `docs/` folder on `master`. So the flow is:

1. edit `content/`
2. `lua main.lua` (rebuild `docs/`) — or just leave `lua run.lua` running
3. commit **both** the `content/` change and the regenerated `docs/*.html`, then push

> Because the built HTML is committed, a content edit produces a large diff (many
> regenerated `.html` files). That's expected. Automating this away (build-on-push via a
> GitHub Action, so you'd only commit `content/`) is a possible future change.

## Add an image

Drop the full-size original anywhere, then optimize it into `docs/assets/images/`:

```sh
tools/optimize-image.sh <src> <name> [photo|shot] [maxwidth]
```

- **photo** (default) — a real photograph → WebP, ~900px wide. e.g.
  `tools/optimize-image.sh ~/Downloads/ibm-pc.jpg ibm-xt photo`
- **shot** — a UI / pixel screenshot → PNG, colour-quantized, ~1000px wide (crisp text).
  e.g. `tools/optimize-image.sh ~/Downloads/editor.png div-games-studio shot`

Use a descriptive `<name>` (no extension — the script picks it). It prints the before→after
size and the exact `![](...)` line to paste into your Markdown. It never upscales and never
writes a file bigger than the source.

## Content notes

- A page is published when its frontmatter has `draft=false`.
- SEO/meta tags come from `meta=true` + `metaDescription` / `metaImg` / `metaUrl` in the
  frontmatter — see `content/makes/achtbaan.md` for the pattern.
- Running notes and decisions live in `PLAN.md` and `TODO.md`.
